### 1 方案1：纯视觉
#### 1.1 6D 卡尔曼滤波：纯角度解耦

原理：

- 6D LOS 滤波把单目相机当作“测角传感器”，只估计惯性系视线单位向量 `lambda_I` 和它的变化率 `lambda_dot_I`。
- 状态向量为：

```text
X = [lambda_x, lambda_y, lambda_z,
     lambda_dot_x, lambda_dot_y, lambda_dot_z]
```

- 观测只使用 bbox 中心点 `(u, v)`，不使用 bbox 宽、高、面积或目标真实尺寸。
- 每个检测框中心先转成相机系单位视线，再结合曝光时间戳查到的历史姿态 `R_IB(t_exposure)`，转换成惯性系观测 `lambda_I_measured`。
- 预测时使用近似常角速度模型：

```text
lambda(k+1) ~= normalize(lambda(k) + lambda_dot(k) * dt)
lambda_dot(k+1) ~= lambda_dot(k)
```

- 更新后强制 `lambda` 归一化，并将 `lambda_dot` 投影到 `lambda` 的垂直平面，避免出现非物理的径向分量。

数据流：

```text
bbox center (u, v)
  -> 相机内参归一化
  -> los_C
  -> R_BC 安装外参
  -> los_B
  -> R_IB(t_exposure) 历史姿态去旋转
  -> lambda_I_measured
  -> 6D Kalman Filter
  -> lambda_I, lambda_dot_I, innovation, quality
```

优势：

- 与相机“测角仪”本质匹配。
- 不受 bbox 尺度抖动直接污染。
- 能输出稳定、平滑、物理一致的 `lambda_dot`。
- 不需要假设目标真实大小。
- 不需要单目绝对测距。
- 可以清晰地用 innovation、协方差和时间戳质量判断该帧是否可信。

劣势：

- 不知道绝对距离。
- 不直接知道闭合速度。
- 若在仿真中固定闭合速度，会导致不同交会角下等效增益变化很大。
- 不能区分“远处高速目标”和“近处低速目标”造成的相似角速度。
- 纯角度信息对接近进程不敏感，尤其在目标近似横穿、面积变化弱时。
- 目标短暂丢失后，只靠角速度模型外推会快速失去可信度。
- 强依赖时间戳对齐；曝光时间和姿态时间错位会直接制造伪 `lambda_dot`。
- 强依赖相机内参、外参和安装刚性；安装角误差会变成系统性 LOS 偏差。
- 目标只占少量像素时，中心点量化误差会放大成角度噪声。

边界限制：

- 目标必须持续可见，且 bbox 中心要稳定；严重遮挡、裁切、误检或 track_id 切换时必须重置或降权。
- 目标距离很远、bbox 只有几个像素时，6D 滤波只能输出低可信角度趋势。
- 目标在画面边缘时，畸变和 bbox 裁切会降低 LOS 可信度。
- 高机动目标会让 `lambda_dot` 快速变化，滤波器需要提高过程噪声，但噪声过大又会放大检测抖动。
- 高速迎近目标虽然角度可能稳定，但风险进程很快；6D LOS 本身无法给出接近时间，必须依赖 TTC、雷达或其他距离/速度来源。
- 纯横穿目标的角速度明显，但闭合趋势可能弱；6D LOS 不能单独判断是否正在接近。
- 飞机自身大角速度运动时，必须依赖 Pixhawk 历史姿态去旋转；姿态缓存缺失时该帧不能用于更新。

#### 1.2 纯视觉推荐融合架构

纯视觉融合架构的核心不是“用 bbox 测距”，而是把相机最擅长的测角能力和 bbox 面积的相对膨胀趋势分开使用。角度通道只回答“目标方向如何变化”，尺度通道只回答“目标是否在快速接近”，二者通过质量门控组合，不互相污染状态。

```text
输入层：
  frame_id + exposure_timestamp
  bbox center (u, v)
  bbox area A
  track_id + track_quality
  Pixhawk historical attitude R_IB(exposure_timestamp)

角度通道：
  输入：bbox 中心点 (u, v)
  处理：曝光时间戳对齐 + R_IB 历史姿态去旋转 + 6D LOS 滤波
  输出：lambda_I, lambda_dot_I, innovation

接近进程通道：
  输入：bbox 面积 A
  处理：面积低通滤波 + 裁切/突变剔除 + 面积导数 + TTC 估计
  输出：TTC, TTC_quality, scale_continuity

融合与门控：
  仅当 track_id 连续、姿态查表成功、LOS innovation 正常、TTC_quality 有效时输出评估量
  任一质量门失败时，只记录日志，不输出受监督候选量
```

受监督评估中可记录如下候选量：

```text
g_eval = K(TTC) * lambda_dot_I
```

其中 `K(TTC)` 只作为仿真和日志分析中的动态增益因子。禁止将该候选量绕过安全状态机直接连接到高风险实飞控制输出。

主要特点：

- 角度与尺度完全解耦：bbox 面积抖动不会通过距离状态污染 `lambda_dot_I`。
- 不依赖目标真实尺寸：TTC 使用面积相对变化，而不是绝对尺寸反推距离。
- 强依赖时间同步：曝光时间戳和 Pixhawk 历史姿态是 LOS 稳定性的前提。
- 强依赖 track 连续性：target_id 变化、短时遮挡或 bbox 裁切都会使 TTC 失效。
- 输出是“质量受控的评估量”：适合日志、仿真和受监督实验，不是无条件控制指令。

适用边界：

- 目标必须在相机视场内保持足够帧数，且 bbox 中心可稳定测量。
- 目标图像尺度需要有可观测变化；远距离小目标只有几像素时，TTC 不可信。
- 目标速度过低或横向通过但几乎不接近时，面积膨胀弱，TTC 会退化。
- 目标速度过高或交会时间过短时，可用帧数不足，面积导数容易被延迟和抖动主导。
- 目标大幅机动、强滚转、姿态剧变或被遮挡时，bbox 面积不再只反映距离变化，TTC 必须降权。
- 背景杂波、曝光变化、运动模糊、压缩拖影和检测框裁切会直接降低尺度通道可信度。

敌方无人机速度与运动限制：

- 匀速或低机动目标：最适合该架构，6D LOS 能稳定给出方向变化，TTC 能提供接近趋势。
- 中等机动目标：角度通道仍可工作，但 TTC 需要更强质量门控，因为目标姿态变化会改变 bbox 面积。
- 高机动目标：纯视觉仍能做短时 LOS 评估，但不能可靠估计接近进程；需要雷达、双目、外部真值或其他距离/速度来源补强。
- 纯横穿目标：初期面积变化弱，TTC 可能显示“接近不明显”；此时只能依赖 LOS 角速度和目标保持在视场内的连续观测。
- 快速迎近目标：面积膨胀明显但时间窗口短，要求高帧率、低延迟和稳定曝光时间戳，否则 TTC 会滞后。

工程边界：

- 纯视觉方案不能保证所有速度和机动包线下都有可靠闭合率估计。
- 不能把 `TTC` 当作真实距离，也不能把 `K(TTC)` 当作实飞控制增益直接使用。
- 当 `TTC_quality` 低、LOS innovation 高、track_id 不连续或姿态缓存失败时，系统必须进入记录/安全状态。

### 2 方案二：机载毫米波雷达无人机探测方案

本项目讨论的毫米波雷达不是人体存在检测、生命体征检测或高度计，而是安装在穿越机/无人机机体上的前向空中目标探测雷达。它的任务是发现并跟踪空中小型无人机，输出距离、径向速度、多普勒、方位/俯仰和轨迹质量，用于非武器化日志、融合评估和安全状态判断。

由于雷达必须挂载在穿越机上，选型优先级应改为：

```text
无人机探测能力 > 机载 SWaP > 视场/刷新率/接口/时间戳 > 价格
```

#### 2.1 雷达方案收益

- 可直接观测空中目标距离和径向速度。
- 弱光、逆光和部分烟尘条件下比可见光稳定。
- 与视觉角度测量互补。
- 可用于验证 TTC 与真实距离/速度趋势是否一致。
- 对 RF 静默目标仍有探测价值。

#### 2.2 雷达方案限制

- 小型空中目标 RCS 小，实际探测距离强依赖目标材料、姿态、雷达截面积和背景杂波。
- 典型 5-7 寸穿越机的有效载荷和供电余量很小，成熟 DAA 雷达通常过重、过耗电。
- 低成本 24GHz/60GHz 人体存在模块、生命体征模块和高度计不适合空中无人机探测。
- TI AWR/IWR 等开发板不是开箱即用无人机探测雷达，需要自研点云处理、聚类、跟踪、虚警抑制和时间同步。
- 真正成熟的机载 DAA / C-UAS 雷达价格高、采购多为 RFQ，且可能有合规、出口管制或频谱许可要求。

#### 2.5 推荐结论

1. 当前 MVP 不挂成熟 DAA 雷达，继续以纯视觉 6D + TTC 为主，预留 `RadarMeasurement` 接口。
2. 若必须做机载雷达数据采集，优先使用 AWR1843BOOST/AOP 或 IWR6843ISK 做研究载荷，只验证时间同步、点云、距离/径向速度质量，不承诺稳定探测无人机。
3. 若平台升级到可承载 800 g、45 W 级载荷，并且预算/合规允许，优先评估 EchoFlight；备选 Fortem TrueView R20/R20i。
4. 低成本存在检测、生命体征和高度计模块从无人机探测候选中删除。

#### 2.6 雷达融合建议

非武器化评估链路中，雷达只进入独立测距/测速通道：

```text
视觉：
  bbox center -> 6D LOS -> lambda_I, lambda_dot_I

视觉尺度：
  bbox area -> TTC -> 接近趋势

雷达：
  range, range_rate, doppler, point cloud -> 距离/径向速度质量评估

融合评估：
  对比 TTC 与 radar range_rate
  对比视觉 LOS 与 radar angle/point cloud
  输出日志、置信度和安全状态
```

工程推荐：

- MVP 仍以纯视觉 6D + TTC 为主，并预留雷达接口。
- 研发阶段优先用 TI AWR/IWR 做机载记录和台架/HIL。
- 工程化无人机探测优先看 EchoFlight / Fortem R20 级别，但必须先确认穿越机载荷、供电和散热余量。
- 地面 C-UAS 雷达只用于外部真值，不作为机载选型。

### 4 比例导引算法流程：三类感知方案后的实施思路

本章说明在三类感知输入之后如何组织比例导引（PNG, Proportional Navigation Guidance）的算法流程：

1. 仅使用 6D 纯角度 LOS 滤波。
2. 使用 6D LOS + Scale Expansion / TTC 的纯视觉融合架构。
3. 使用视觉 + 机载毫米波雷达测量方案。

这里的“实施”指仿真、回放、台架、HIL 和受监督评估中的算法流程设计，不表示绕过安全状态机直接执行高风险自主控制。

---

#### 4.1 PNG 的理论基础

比例导引的核心思想是：如果拦截双方最终会在同一点相遇，那么视线方向在惯性空间中应逐渐趋于稳定。换句话说，理想碰撞航线上，视线角速度趋近于零。

经典 PNG 常写为：

```text
a_n = N * V_c * omega_LOS
```

其中：

- `a_n`：期望法向机动量，方向应与视线垂直。
- `N`：导引比或导航常数。
- `V_c`：闭合速度或其等效量。
- `omega_LOS`：视线角速度。

在向量形式下，常用：

```text
omega_LOS = lambda x lambda_dot
```

其中：

- `lambda` 是惯性系视线单位向量。
- `lambda_dot` 是视线单位向量变化率。

工程上要特别注意：PNG 最敏感的变量不是距离，而是 `lambda_dot` 和闭合进程判断。如果 `lambda_dot` 被噪声污染，导引输出会振荡；如果闭合速度或接近进程估计错误，等效导引比会失真。

因此后续三类流程的差别，本质在于：

- 谁提供稳定的 `lambda_dot`。
- 谁提供 `V_c` 或接近进程替代量。
- 谁提供质量门控，决定当前输出是否可信。

---

#### 4.2 基于 6D 纯角度 LOS 的 PNG 流程

##### 4.2.1 输入与输出

输入：

- bbox 中心点 `(u, v)`。
- 图像曝光时间戳。
- Pixhawk 历史姿态 `R_IB(t_exposure)`。
- 相机内参和安装外参 `R_BC`。
- 自机速度和飞控健康状态。

6D LOS 输出：

- `lambda_I`。
- `lambda_dot_I`。
- `omega_LOS = lambda_I x lambda_dot_I`。
- innovation、协方差、时间戳质量。

PNG 需要的额外量：

- 闭合速度 `V_c` 或等效闭合进程指标。

6D 纯角度方案本身不提供 `V_c`，这是它用于 PNG 时的最大短板。

##### 4.2.2 算法流程

```text
1. 视觉检测得到 bbox center。
2. 用曝光时间戳查询历史姿态。
3. 将 bbox center 转为惯性系视线观测 lambda_I_measured。
4. 6D LOS 滤波输出 lambda_I 和 lambda_dot_I。
5. 计算 omega_LOS = lambda_I x lambda_dot_I。
6. 进行质量门控：
   - track_id 是否连续
   - 姿态查表是否成功
   - LOS innovation 是否正常
   - lambda_dot 是否超出物理合理范围
7. 若质量通过，则进入 PNG 评估层。
8. 若质量失败，则只记录日志，不输出导引候选量。
```

##### 4.2.3 闭合速度处理

纯 6D 角度方案不能可靠估计闭合速度。工程上有三种处理方式：

- 使用场景假设闭合速度：适合仿真或固定测试条件，但跨场景泛化差。
- 使用自机速度投影近似：只能得到自机相对视线方向的贡献，不能知道目标速度。
- 不计算真实 `V_c`，只把 `omega_LOS` 作为角速度风险指标：最安全，适合日志和评估。

不建议在真实复杂场景中把 `V_c` 写死后直接用于高风险闭环。原因是不同交会角下真实闭合速度差异很大，固定 `V_c` 会导致等效导引比不稳定。

##### 4.2.4 主要特点

- 优点：角速度估计最稳定，结构简单，不依赖目标真实尺寸。
- 缺点：缺少接近进程，不知道距离和闭合速度。
- 适合：验证 LOS 稳定性、评估目标方向变化、做低风险仿真输入。
- 不适合：单独承担完整 PNG 增益调度。

##### 4.2.5 工程技巧

- `lambda_dot` 必须投影到 `lambda` 的垂直平面，避免非物理径向分量。
- `omega_LOS` 输出前应做轻度低通或限幅，但不能用过强滤波制造明显相位滞后。
- 目标丢失、track_id 切换、bbox 裁切时，滤波器应重置或降权。
- 姿态时间戳对齐失败时，宁可丢弃该帧，也不要用当前姿态替代历史姿态。
- 6D LOS 输出应始终带质量评分，而不是只输出一个数值。

#### 4.3 基于 6D LOS + TTC 融合架构的 PNG 流程

##### 4.3.1 架构思想

6D LOS + TTC 融合架构将 PNG 所需变量拆成两条通道：

```text
角度通道：
  6D LOS -> lambda_I, lambda_dot_I, omega_LOS

接近进程通道：
  bbox area -> A_filt, A_dot_filt -> TTC, TTC_quality
```

两条通道的分工：

- 6D LOS 负责“往哪个方向变化”。
- TTC 负责“接近进程是否正在加快”。

这样做的好处是，bbox 面积噪声不会污染 `lambda_dot`，而 `lambda_dot` 又不需要承担测距职责。

##### 4.3.2 TTC 与 PNG 增益的关系

经典 PNG 使用 `V_c`，但纯视觉难以可靠得到 `V_c`。TTC 可以作为接近进程的替代指标，用于调节导引评估量的动态增益。

可表达为：

```text
g_eval = K(TTC) * omega_LOS
```

或使用 `lambda_dot_I` 形式：

```text
g_eval = K(TTC) * lambda_dot_I
```

这里 `K(TTC)` 不是实飞控制增益，而是仿真、日志和受监督评估中的动态权重。通常思路是：

- TTC 较大：说明接近还不紧迫，增益应较低，避免对噪声过度敏感。
- TTC 中等：进入主要评估区间，增益平滑增加。
- TTC 很小：说明时间窗口很短，但此时视觉也最容易裁切、模糊和失效，必须同时检查质量门控。

工程上不建议让 `K(TTC)` 突变。应使用连续函数或分段平滑函数，并对输出变化率做限制，避免由 TTC 抖动带来导引候选量振荡。

##### 4.3.3 算法流程

```text
1. 检测和跟踪输出 bbox center、bbox area、track_id。
2. 角度通道：
   - bbox center -> 时间戳对齐 -> 历史姿态去旋转 -> 6D LOS
   - 输出 lambda_I、lambda_dot_I、omega_LOS 和 LOS 质量
3. 尺度通道：
   - bbox area -> 低通滤波
   - 面积突变、裁切、track_id 切换检测
   - 计算 A_dot_filt
   - 估计 TTC 和 TTC_quality
4. 融合门控：
   - LOS quality 必须有效
   - TTC_quality 必须有效
   - track_id 必须连续
   - 视觉延迟和姿态缓存必须在允许范围内
5. 形成导引评估量：
   - g_eval = K(TTC) * omega_LOS
6. 输出到日志、仿真或受监督评估层。
7. 任一质量门失败时，不输出候选量，只记录失败原因。
```

##### 4.3.4 主要特点

- 比纯 6D LOS 更完整：补充了接近进程。
- 比 8D bbox 测距 EKF 更稳定：尺度噪声不进入 LOS 主状态。
- 不需要目标真实尺寸：TTC 用面积相对变化。
- 对检测框质量要求高：面积导数是高噪声量。

##### 4.3.5 边界限制

- 目标远距离过小：面积变化不明显，TTC 不可信。
- 目标纯横穿：角速度明显但面积变化弱，TTC 会退化。
- 目标高速迎近：TTC 明显但可用帧数少，对延迟极敏感。
- 目标姿态剧变或强滚转：bbox 面积变化可能来自姿态而不是距离。
- bbox 被画面边缘裁切：TTC 必须无效。
- 目标被遮挡或 detector 输出跳变：必须冻结或重置尺度通道。

##### 4.3.6 工程技巧

- 面积滤波使用两级处理：先剔除异常值，再低通滤波。
- TTC 不直接用单帧差分，应使用滑动窗口或鲁棒斜率估计。
- `K(TTC)` 应有上下限和变化率限制。
- TTC 和 LOS 的时间戳必须一致，不能用不同帧的面积和角度混合。
- 质量评分要分开记录：`LOS_quality`、`TTC_quality`、`track_quality`、`timestamp_quality`。
- 在接近末段，如果 bbox 快速变大并出现裁切，不能继续信任 TTC，应进入视觉失效处理或安全状态。

#### 4.4 基于雷达测量方案的 PNG 流程

##### 4.4.1 架构思想

雷达方案的核心优势是提供视觉缺失的距离和径向速度：

```text
视觉：
  lambda_I, lambda_dot_I, target_id, class/track quality

雷达：
  range, range_rate, doppler, azimuth, elevation, SNR, radar_track_quality

融合：
  几何一致性检查 + 时间对齐 + 质量门控
```

在这种架构中：

- 视觉负责目标识别、角度连续性和 LOS。
- 雷达负责距离、径向速度和多普勒。
- Pixhawk 负责姿态、速度、时间基准和飞控状态。

##### 4.4.2 PNG 变量来源

雷达融合后，PNG 变量可以更接近经典形式：

```text
omega_LOS = lambda_I x lambda_dot_I
V_c ~= -range_rate
```

其中：

- `range_rate < 0` 通常表示目标距离在减小。
- `V_c` 应使用经过质量门控和滤波后的径向闭合速度。
- 若雷达只给点云而不给 track，需要先做聚类、跟踪和目标关联。

##### 4.4.3 雷达 EKF 融合层

雷达输出通常是极坐标量，例如：

```text
z_radar = [range, azimuth, elevation, range_rate]
```

而 PNG 更希望得到连续、稳定、带质量评估的相对状态。因此雷达方案中建议增加一个扩展卡尔曼滤波（EKF）层，用于把雷达极坐标测量转换为目标相对位置、相对速度和闭合速度估计。

推荐状态定义：

```text
X_r = [p_x, p_y, p_z,
       v_x, v_y, v_z]
```

其中：

- `p` 是目标相对自机的位置，建议表达在惯性系或本地 NED 系。
- `v` 是目标相对自机的速度。
- 如果雷达坐标先在雷达坐标系中得到，需要通过雷达外参、机体姿态和时间戳转换到统一坐标系。

预测模型可采用常速度模型：

```text
p(k+1) = p(k) + v(k) * dt
v(k+1) = v(k)
```

雷达观测模型是非线性的：

```text
range       = norm(p)
azimuth     = atan2(p_y, p_x)
elevation   = atan2(-p_z, sqrt(p_x^2 + p_y^2))
range_rate  = dot(p, v) / norm(p)
```

这就是需要 EKF 的原因：状态是笛卡尔相对位置/速度，而雷达测量是距离、角度和径向速度，两者之间存在明显非线性。

EKF 在该方案中的职责：

- 平滑 `range` 和 `range_rate`，避免单帧雷达跳变直接影响 PNG 评估量。
- 输出相对位置 `p_rel` 和相对速度 `v_rel`。
- 给出闭合速度估计：

```text
lambda_r = p_rel / norm(p_rel)
V_c = -dot(lambda_r, v_rel)
```

- 提供协方差、创新量和雷达测量质量，用于融合门控。
- 处理雷达短时丢失时的 coast，但 coast 时间必须有限。

视觉与雷达的关系：

- 视觉 6D LOS 仍作为角度主通道，因为相机角分辨率通常高于小型机载雷达。
- 雷达 EKF 主要提供 `range`、`range_rate`、`V_c` 和相对运动连续性。
- 若雷达角度与视觉 LOS 方向一致，可提高融合可信度。
- 若二者角度长期不一致，应认为目标关联失败，而不是强行融合。

雷达 EKF 的质量门控：

- 距离创新过大：拒绝该雷达更新。
- 角度创新过大：降低关联可信度或重新关联。
- 径向速度突变：检查多普勒混叠、点云跳变或目标切换。
- SNR 过低：降低雷达测量权重。
- 点云数量过少或聚类不稳定：不更新 EKF，只预测。
- 雷达时间戳超窗：拒绝更新。

工程技巧：

- EKF 初始化不要只靠单帧雷达，最好要求连续多帧稳定点云或 track。
- 初始速度可由连续 range/range_rate 估计，不要给过小协方差。
- 雷达角度噪声通常大于视觉角度噪声，融合时不要让雷达角度覆盖视觉 LOS。
- `range_rate` 很有价值，但容易受目标关联错误影响，必须做连续性检查。
- 过程噪声要允许目标有适度机动，但不能大到把虚警也跟踪成真实目标。
- EKF 输出应始终带 `radar_state_quality`，低质量时不得参与 PNG 评估。

##### 4.4.4 算法流程

```text
1. 视觉通道输出：
   - lambda_I
   - lambda_dot_I
   - visual_track_id
   - LOS_quality

2. 雷达通道输出：
   - range
   - range_rate
   - azimuth/elevation
   - SNR
   - radar_track_quality

3. 雷达 EKF 更新：
   - 将雷达测量转换到统一时间基准和坐标系
   - 使用 EKF 预测/更新相对位置 p_rel 和相对速度 v_rel
   - 输出 filtered range、filtered range_rate、V_c、radar_state_quality

4. 时间对齐：
   - 将视觉测量、雷达测量和 Pixhawk 姿态统一到同一单调时钟
   - 超过时间窗的测量不得融合

5. 空间关联：
   - 比较视觉 LOS 方向与雷达角度/点云方向
   - 比较视觉 TTC 趋势与雷达 range_rate 趋势
   - 不一致时降低融合可信度或拒绝输出

6. 变量生成：
   - omega_LOS = lambda_I x lambda_dot_I
   - V_c = EKF 输出的 filtered closing speed
   - range_quality、range_rate_quality、association_quality、radar_state_quality

7. PNG 评估：
   - 使用质量门控后的 omega_LOS 和 V_c 形成导引评估量
   - 输出到仿真、日志或受监督评估层

8. 异常处理：
   - 雷达目标丢失、SNR 低、点云跳变、视觉雷达关联失败时，不输出候选量
```

##### 4.4.5 主要特点

- 比纯视觉更接近完整 PNG 所需变量。
- 雷达能提供距离和径向速度，显著降低闭合进程不确定性。
- 视觉能提供目标类别和更高角分辨率，补足雷达分类能力不足。
- 融合质量取决于视觉目标和雷达目标能否正确关联。
- EKF 能把跳变的极坐标测量变成连续的相对位置/速度估计。

##### 4.4.6 边界限制

- 小型无人机 RCS 小，雷达探测距离和稳定性强依赖目标姿态、材料和背景杂波。
- 雷达角分辨率通常不如相机，单独用雷达角度做 LOS 可能较粗。
- 机载雷达有重量、功耗、散热和安装遮挡问题，普通 5-7 寸穿越机难以承载成熟 DAA 雷达。
- 多目标环境中，视觉 track 和雷达 track 的关联是关键难点。
- 雷达点云可能受地面、建筑、桨叶、机体结构和电磁噪声影响产生虚警。
- 雷达和视觉时间戳不同步时，融合结果会出现虚假相对运动。
- EKF 不能解决错误关联问题；如果雷达 track 对错目标，滤波只会稳定地跟错目标。

##### 4.4.7 工程技巧

- 雷达测量进入融合前应先做 SNR、速度连续性、距离连续性和点云数量门控。
- `range_rate` 应滤波，但不能过度平滑，否则会引入闭合速度滞后。
- 视觉 LOS 与雷达角度应做门限关联，超过角度门限则拒绝融合。
- 视觉 TTC 与雷达 `range / -range_rate` 趋势可互相校验。
- 雷达安装位置应前向无遮挡，避免碳纤维、电池、金属件和桨叶进入主要波束。
- 雷达坐标系外参必须标定，并和相机外参、机体系保持一致。
- 融合层应输出 `association_quality`，不能只输出融合后的单一数值。

#### 4.5 三种 PNG 流程对比

| 方案 | LOS 角速度来源 | 闭合进程来源 | 主要优势 | 主要短板 | 推荐用途 |
| --- | --- | --- | --- | --- | --- |
| 6D 纯角度 LOS | bbox center + 姿态去旋转 + 6D 滤波 | 无可靠来源，只能假设或记录角速度风险 | 最稳定、最简单、不受 bbox 尺度污染 | 不知道距离和闭合速度 | LOS 稳定性验证、日志、仿真输入 |
| 6D LOS + TTC | 6D LOS | bbox 面积膨胀估计 TTC | 单目条件下最均衡，不依赖目标真实尺寸 | TTC 对 bbox jitter 和姿态变化敏感 | 纯视觉 MVP、回放、HIL、受监督评估 |
| 视觉 + 雷达 | 视觉 6D LOS 为主，雷达角度辅助 | 雷达 range/range_rate | 最接近完整 PNG 变量，闭合速度可信度更高 | 硬件重量/功耗/成本高，目标关联复杂 | 大载重平台、机载雷达数据采集、融合验证 |

#### 4.6 总体工程建议

建议按以下顺序推进，而不是一步到位：

1. 先完成 6D LOS：确保 `lambda_dot_I` 稳定、时间戳正确、姿态去旋转有效。
2. 再加入 TTC：验证 bbox 面积膨胀是否能稳定反映接近趋势。
3. 再做 PNG 评估量：只在日志、仿真和受监督评估层使用。
4. 最后加入雷达：用真实 range/range_rate 校验 TTC 和闭合速度假设。

工程门控原则：

- 没有稳定 `lambda_dot`，不做 PNG 评估。
- 没有可信闭合进程，不做动态增益评估。
- 没有时间戳一致性，不融合视觉和雷达。
- 没有目标关联质量，不输出融合候选量。
- 任一质量门失败，进入记录/安全状态。

最终推荐架构：

```text
基础层：
  Pixhawk EKF + 时间戳同步 + 姿态 Ring Buffer

视觉角度层：
  bbox center -> 6D LOS -> lambda_I, lambda_dot_I

接近进程层：
  纯视觉：bbox area -> TTC
  雷达增强：range/range_rate -> V_c

融合质量层：
  track_quality + LOS_quality + TTC_quality/radar_quality + timestamp_quality

PNG 评估层：
  仅在质量门控通过时形成导引评估量

安全层：
  日志、仿真、HIL、受监督输出、安全退出
```
