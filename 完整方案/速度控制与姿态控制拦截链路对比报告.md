# 速度控制与姿态控制拦截链路对比报告

## 1. 目的

本文对比当前捷联视觉 PNG 中两条控制链路：

- 速度链路：`guidance_output_mode=velocity_bias`，`px4_command_mode=velocity_yaw_rate`
- 姿态链路：`guidance_output_mode=accel_attitude`，`px4_command_mode=mavlink_attitude`

分析目标是解释为什么当前实验中姿态控制拦截效果不如速度控制，并给出下一步改进计划。

## 2. 实验依据

速度链路使用日志：

```text
logs/yolo_sitl_ttc_vm/yolo_sitl_*_legacy_retest_20260625_005552_r*_h20.csv
```

姿态链路使用日志：

```text
logs/yolo_sitl_ttc_vm/yolo_sitl_*_attitude_ttc_vm_20260623_092510_r*_h20.csv
```

两批均为 PX4 SITL + Actor 目标，目标使用 `Quadrotor1` actor，视觉检测为 YOLOv8 + ByteTrack。距离为 50m、60m、70m、80m、90m、100m，高度差为 20m。

## 3. 两条控制链路的思路

### 3.1 共同的视觉 PNG 前端

两条链路共用同一套视觉导引前端：

1. YOLOv8 + ByteTrack 输出目标 `bbox center`、`bbox area`、`track_id`。
2. `bbox center` 由相机内参转为相机坐标系 LOS。
3. 利用机体姿态和相机外参，将 LOS 转到惯性系：

```text
lambda_I = normalize(R_IB * R_BC * lambda_C)
```

4. 相邻帧 LOS 差分得到 LOS 角速度：

```text
lambda_dot = d(lambda_I) / dt
omega_LOS = lambda_I x lambda_dot
```

5. 导引律生成 PNG 需用加速度：

```text
a_cmd = gain * (omega_LOS x lambda_I)
n_cmd_g = ||a_cmd|| / g
```

其中：

- `lambda_I`：惯性系视线单位向量。
- `omega_LOS`：视线角速度。
- `gain`：导引增益。TTC 模式由 TTC 调度，VM 模式为 `N * V_m`。
- `a_cmd`：导引层希望产生的横向需用加速度。
- `n_cmd_g`：导引层需用过载。

### 3.2 速度链路：PNG 修正直接进入速度 setpoint

速度链路的核心思想是把 PNG 修正转成速度指令的一部分：

```text
v_cmd = v_los_base + v_png_bias
```

其中 `v_los_base` 沿 LOS 指向目标，大小受 `speed_cap` 限制；`v_png_bias` 来自 LOS 角速度修正。程序最终向 AirSim/PX4 发送速度和 yaw-rate：

```text
moveByVelocity / MAVLink velocity setpoint + yaw_rate
```

它的特点是：

- 控制目标是速度，不直接要求某个姿态角。
- PX4/AirSim 内部速度闭环负责把速度误差转换为姿态、推力和电机输出。
- PNG 修正具有积分型效果，容错更强。
- 即使短时 LOS 或姿态有误差，速度闭环仍会持续把拦截机拉向目标。

这条链路更像工程上的“视觉 LOS 速度追踪器”，不是严格的过载指令 PNG。

### 3.3 姿态链路：PNG 加速度反解为姿态四元数和油门

姿态链路先生成 PNG 需用加速度，然后叠加沿 LOS 的速度保持项：

```text
a_speed_hold = K_v * (v_ref - v_current)
a_control = clip(a_cmd + a_speed_hold)
```

随后把 `a_control` 按期望 yaw 旋转到水平坐标系，并反解为 roll/pitch：

```text
a_yaw_body = R_z(yaw_sp)^T * a_control
roll_sp  = atan2(...)
pitch_sp = atan2(...)
```

roll/pitch 会受到 `attitude_max_tilt_deg` 限幅。然后构造姿态四元数：

```text
q_sp = quat(R_z(yaw_sp) * R_y(pitch_sp) * R_x(roll_sp))
```

油门由质量、最大推力和垂向加速度需求估算：

```text
thrust = mass * (g - a_z) / (max_total_thrust * cos(roll) * cos(pitch))
```

最终通过 MAVLink `SET_ATTITUDE_TARGET` 发送：

```text
attitude quaternion + normalized thrust
```

这条链路更接近真实飞控：导引输出的是过载/加速度，再由姿态和油门实现。但它对模型和控制响应的要求明显更高。

## 5. 为什么姿态控制效果更差

### 5.1 速度链路把难题交给 PX4 速度闭环

速度链路只要求 PX4 跟踪速度 setpoint。PX4 内部会自动处理：

- 姿态角生成
- 推力分配
- 速度误差积分
- 加速度/倾角限制

因此它对质量、推力、姿态响应建模误差不敏感。即使视觉 LOS 有短时抖动，速度指令仍会把拦截机持续拉向目标。

### 5.2 姿态链路直接暴露动力学和限幅问题

姿态链路把 PNG 加速度反解为 roll、pitch 和 thrust。这个过程要求：

- 质量准确。
- 最大总推力准确。
- hover thrust 准确。
- PX4 姿态环响应足够快。
- AirSim GenericQuad 和 PX4 SITL 动力学一致。
- 小角度近似在当前机动范围内仍然可用。

日志显示 roll/pitch setpoint 经常接近或达到 `25 deg` 限幅，说明导引需求已经超过当前姿态控制能够实现的范围。一旦指令饱和，PNG 需用过载就不能被真实实现。

### 5.3 姿态滞后会破坏捷联相机视场

捷联相机固定在机体上。姿态控制一旦有 yaw/pitch/roll 滞后，影响不只是轨迹误差，还会直接影响相机指向。

这会造成连锁反应：

```text
姿态滞后
-> 目标偏离图像中心
-> YOLO/ByteTrack 检测连续性下降
-> LOS/KF 降级或外推
-> PNG 需用加速度不再可靠
-> 脱靶或错过碰撞窗口
```

速度链路虽然也有机体姿态变化，但它对位置闭环更“钝化”，因此在检测连续性不完美时更容易保持接近目标。

### 5.4 姿态链路的控制频率和视觉频率不匹配

当前 YOLO 闭环约 8 Hz。姿态控制需要在低频视觉输入下生成较高动态的姿态/油门指令。末端相对角速度大，8 Hz 下每帧之间目标位置变化明显，姿态控制又存在响应滞后，容易错过末端 0.1-0.4 s 的碰撞窗口。

速度链路相当于把低频视觉输出变成较平滑的速度目标，PX4 的速度控制器继续在内部高频运行，因此表现更稳。

### 5.5 姿态链路中的 speed_hold 会改变纯 PNG

姿态模式不是单纯下发 `a_cmd`，而是：

```text
a_control = a_cmd + a_speed_hold
```

`a_speed_hold` 用于保持沿 LOS 的追击速度，这是必要的工程项，但它会改变纯 PNG 加速度方向。若速度保持项过强，会出现两个问题：

- 横向 PNG 修正被沿 LOS 的速度保持“稀释”。
- 姿态角被速度保持项占用，导致可用于横向修正的倾角余量不足。

这也是姿态模式需要单独调参的原因，不能直接复用速度模式参数。

## 6. 对两种链路的评价

### 6.1 速度链路优点

- 命中率更高。
- 参数更少，调试更快。
- 对飞行器质量、推力模型不敏感。
- 对低频视觉和短时丢检更鲁棒。
- 适合作为算法评价基线和视觉闭环调试工具。

### 6.2 速度链路缺点

- 不是严格的过载式 PNG。
- PNG 修正进入速度指令后具有积分效应，理论解释不如姿态/过载链路清晰。
- 与真实飞控实现链路有差距。
- 无法直接验证“需用过载是否被机体真实实现”。

### 6.3 姿态链路优点

- 更接近真实导弹/无人机控制结构：PNG 输出过载，底层姿态/推力实现过载。
- 可以明确记录 `a_cmd`、`n_cmd_g`、`roll_sp`、`pitch_sp`、`thrust` 和实际姿态响应。
- 适合后续接入 HIL 或真实 PX4。
- 能暴露真实动力学限制和控制带宽问题。

### 6.4 姿态链路缺点

- 当前命中率较低。
- 对质量、推力、hover thrust、倾角限幅、PX4 参数非常敏感。
- 低频视觉下容易把目标甩出画面。
- 姿态跟踪滞后会直接污染捷联相机 LOS。
- 需要单独设计末端视场保持和姿态响应调参。

## 7. 下一步计划

### 7.1 第一阶段：校准姿态/推力模型

目标：确认姿态 setpoint 能否产生期望加速度。

计划：

1. 单独做阶跃测试，不跑拦截。
2. 分别下发固定 roll/pitch/thrust。
3. 记录实际速度、加速度、姿态响应。
4. 拟合以下参数：
   - `vehicle_mass_kg`
   - `vehicle_max_total_thrust_n`
   - `attitude_hover_thrust`
   - `attitude_thrust_gain`
   - PX4 姿态响应延迟
5. 输出“姿态角 -> 实际水平加速度”的标定曲线。

验收标准：

- 固定 roll/pitch 下，实际水平加速度和理论值误差小于 20%。
- hover thrust 下高度漂移可控。
- 姿态阶跃响应延迟和超调可量化。

### 7.2 第二阶段：降低姿态链路激进程度

目标：减少 roll/pitch 饱和和末端甩视场。

建议参数方向：

```text
attitude_max_tilt_deg: 25 -> 20 或 22
attitude_speed_hold_gain: 1.2 -> 0.6~0.9
attitude_speed_hold_max_accel_mps2: 6.0 -> 3.0~4.0
attitude_total_accel_limit_mps2: 18.0 -> 10.0~14.0
max_guidance_accel_mps2: 15.0 -> 8.0~12.0
```

调参原则：

- 先让姿态不饱和，再提高机动性。
- 先保证目标在画面里，再追求最短拦截时间。
- 不直接追速度链路的 `speed_cap=10m/s` 效果。

### 7.3 第三阶段：加入姿态链路专用 frame-centering

目标：捷联相机先看住目标，再做强 PNG。

建议逻辑：

```text
if image_error_ratio > guard_threshold:
    reduce speed_hold
    reduce lateral PNG gain
    prioritize yaw/pitch centering
    freeze or smooth roll command
```

需要新增或强化的日志字段：

- `frame_centering_attitude_active`
- `attitude_png_gain_scale`
- `attitude_speed_hold_scale`
- `image_error_ratio`
- `roll_sp_limited`
- `pitch_sp_limited`
- `attitude_tracking_error_deg`

### 7.4 第四阶段：分离 yaw 看目标和 roll/pitch 机动

当前姿态链路 yaw setpoint 有约 `11.25 deg` P95 误差。对于捷联相机，yaw 控制应该优先保证目标在图像中心。

建议：

1. yaw 用图像中心误差闭环，而不是只依赖 LOS 水平投影。
2. 末端降低横向 roll 机动，让 yaw/pitch 优先看住目标。
3. 如果 yaw 响应慢，改用 `mavlink_body_rate` 中的 `r_cmd` 直接控制 yaw-rate。

### 7.5 第五阶段：重测姿态链路

推荐测试矩阵：

|阶段|距离|导引|目标|
|---|---|---|---|
|标定后 smoke|50m, 60m|TTC|确认姿态能闭环|
|中距验证|70m, 80m|TTC|检查视场保持|
|VM 对照|50m, 70m, 90m|VM|验证固定 VM 增益|
|完整测试|50-100m|TTC + VM|与速度链路比较|

每次测试必须保存：

- `attitude_roll_sp_deg` / `roll_deg`
- `attitude_pitch_sp_deg` / `pitch_deg`
- `attitude_yaw_sp_deg` / `body_yaw_deg`
- `attitude_thrust`
- `a_cmd_*`
- `attitude_control_accel_*`
- `image_error_ratio`
- `bbox_area_ratio`
- `detected`
- `valid`
- `reject_reason`

## 8. 当前结论

速度控制当前更适合做稳定拦截基线，因为它利用 PX4 速度闭环吸收了大量动力学误差和低频视觉误差。

姿态控制更符合“PNG 输出过载，飞控实现过载”的理论结构，但当前还没有完成动力学和姿态响应标定。日志已经证明姿态指令确实被生成并下发，但实际姿态存在明显滞后和限幅，导致目标容易偏离捷联相机视场，进而引发检测断续、LOS 降级和末端脱靶。

下一步不应直接继续扩大姿态控制测试范围，而应先做姿态/推力标定、降低姿态链路激进程度，并加入姿态链路专用 frame-centering。完成这些之后，再重新比较姿态 PNG 与速度 PNG 的拦截效果。
