# 比例导引到控制链路说明

本文整理当前工程里“比例导引如何变成无人机控制指令”的实际实现。这里的结论先放在前面：

- 当前程序最终下发给 AirSim/PX4 的仍是 `NED 速度指令 v_cmd` 和 `yaw_rate`。
- 新增 `--guidance-output-mode accel_integral` 后，比例导引先生成真实的加速度/过载需求 `a_cmd`，再按仿真步长积分成速度 setpoint。
- 旧的 `--guidance-output-mode velocity_bias` 保留原逻辑：把比例导引修正量直接当作速度修正叠加到 `v_cmd`。
- `n_cmd_g` 是由 `a_cmd` 换算得到的需用过载；旧日志里的 `g_eval` 只作为导引评估量保留。
- 无人机的姿态、推力、电机输出由 AirSim 或 PX4 的内环控制器继续完成。

## 1. 测量到视线

捷联视觉链路里，程序先从检测框得到像面中心，再把像素点转成相机坐标系中的射线，最后结合相机外参和机体姿态旋转到惯性系。

得到的核心量有两个：

- `lambda_I`：惯性系中的 LOS 单位向量
- `omega_LOS`：LOS 角速度

相关实现可见：

- [`examples/run_airsim_strapdown_vision_png.py`](</home/linux/Documents/PNG/examples/run_airsim_strapdown_vision_png.py:2404>)

## 2. 比例导引的导引量

当前工程里，比例导引的核心加速度需求可以写成：

```text
a_cmd = gain * (omega_LOS x lambda_I)
```

其中：

- `omega_LOS x lambda_I = lambda_dot_I`，表示 LOS 单位向量的时间导数
- `omega_LOS x lambda_I` 是一个垂直于视线方向的法向加速度方向
- `gain` 是比例导引增益
- `a_cmd` 是加速度需求，单位 `m/s^2`
- `n_cmd_g = ||a_cmd|| / 9.80665` 是需用过载，单位 `g`

在固定 `V_m` 的模式下，增益通常取：

```text
gain = N * V_m
```

而工程里常把：

```text
V_m = speed_ratio * intruder_speed
```

因此固定 `V_m` 模式可以理解成：

```text
a_cmd = N * V_m * (omega_LOS x lambda_I)
```

TTC 模式下，TTC 主要用于调度 `gain`，不是把控制直接切换成另一种完全不同的姿态控制。工程里推荐使用 `--ttc-soft-guidance`，即主导引仍是 LOS + `V_m`，TTC 只改变增益和末端状态触发。

相关实现可见：

- [`vision_guidance/png_eval.py`](</home/linux/Documents/PNG/vision_guidance/png_eval.py:1>)

## 3. `correction` 的含义

旧速度偏置模式里的：

```text
correction = gain * (omega_LOS x lambda_I)
```

表示“根据视线转动趋势，对速度指令增加的修正量”。

它的作用不是直接控制姿态，而是让拦截机不只沿当前视线直飞，而是提前往更有利的方向修正。

可以把它理解为：

- `base`：朝目标当前视线方向飞
- `correction`：根据目标视线变化，向侧向或高度方向提前补偿

最终合成的不是加速度指令，而是速度指令：

```text
v_cmd = base + correction
```

新增加速度积分模式不再把 `correction` 直接加到速度上，而是先计算：

```text
a_cmd = gain * (omega_LOS x lambda_I)
```

然后执行一阶积分：

```text
v_cmd = current_velocity + a_cmd * dt
```

最后再经过 `speed_cap`、最小速度、横向/纵向限幅、frame_centering 和 terminal extrapolation 等约束。这样日志中的 `a_cmd_norm_mps2` / `n_cmd_g` 才是导引律本身要求的需用加速度/过载，而不是速度指令差分反推出来的等效量。

## 4. `speed_cap` 的含义

`speed_cap` 是外环给拦截机设定的速度上限，单位是 `m/s`。

通常它由下面的式子得到：

```text
speed_cap = speed_ratio * intruder_speed
```

它有三个作用：

1. 作为沿视线方向的基础速度

```text
base = speed_cap * lambda_I
```

2. 限制最终速度指令的模长

如果合成后的 `v_cmd` 太大，会被缩放回 `speed_cap` 以内。

3. 决定 `V_m` 型比例导引的增益规模

固定 `V_m` 模式里，`speed_cap` 实际上就是 `V_m` 的工程近似。

所以 `speed_cap` 可以理解为：

```text
拦截机外环允许的最大追击速度
```

它不是无人机真实速度的保证值，只是期望速度上限。真实速度还会受飞控内环、动力学、限幅和时延影响。

## 5. 最终如何控制无人机

当前程序下发给无人机的实际指令主要有两个：

- `v_cmd = [vx, vy, vz]`，单位 `m/s`
- `yaw_rate_cmd_deg_s`，单位 `deg/s`

在 AirSim 调用里，它们通过速度接口下发，例如：

```python
moveByVelocityAsync(
    vx, vy, vz,
    drivetrain=MaxDegreeOfFreedom,
    yaw_mode=YawMode(is_rate=True, yaw_or_rate=yaw_rate_deg_s)
)
```

在 PX4 MAVLink Offboard 模式下，则发送：

```text
SET_POSITION_TARGET_LOCAL_NED
```

并且只使用速度和 yaw-rate，位置和加速度字段被忽略。

## 6. 控制量和物理含义

| 量 | 含义 | 是否直接发给无人机 |
|---|---|---|
| `g_eval` | 具有加速度型意义的导引评估量 | 否 |
| `correction` | 速度指令的修正分量 | 否，先参与合成 |
| `speed_cap` | 速度上限 / `V_m` 近似值 | 否 |
| `v_cmd` | 本地 NED 速度指令 | 是 |
| `yaw_rate_cmd_deg_s` | 偏航角速度指令 | 是 |
| roll / pitch / thrust | 真实姿态和推力 | 否，由飞控内环生成 |

## 7. 一句话总结

当前比例导引链路是：

```text
检测 / 真值
  -> LOS 与 LOS 角速度
  -> PNG 导引量 g_eval
  -> 速度修正 correction
  -> 速度指令 v_cmd + yaw_rate
  -> AirSim / PX4 内环
  -> 姿态 / 推力 / 电机
```

也就是说，这套实现本质上是“外环比例导引生成速度和偏航角速度，内环负责把它变成真正的飞行控制”。
