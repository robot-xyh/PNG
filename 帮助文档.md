# PNG 项目后续运行帮助文档

本文档说明当前项目在本机继续运行、验证和调试的流程。当前代码定位是纯视觉比例导引算法的仿真评估框架：目标检测先由 AirSim 内置 `detect` 接口提供，算法只使用检测框，不读取入侵无人机真实位置、真实速度或相对真值。

## 1. 当前项目内容

项目根目录：`/home/linux/Documents/PNG`

主要文件：

- `vision_guidance/`：纯视觉导引算法模块。
- `examples/run_synthetic.py`：不依赖 AirSim 的合成检测框示例。
- `examples/run_airsim_blocks.py`：AirSim Blocks 示例，使用 AirSim 内置检测框。
- `tests/`：单元测试。
- `run_blocks_nvidia.sh`：强制使用 NVIDIA 显卡启动 Blocks 的脚本。
- `拦截方案.md`：理论方案文档。
- `README.md`：英文简要说明。

当前实现的核心链路：

1. 检测框中心点 `(u, v)` 转换为相机视线向量。
2. 结合图像曝光时间对应的姿态矩阵，转换到惯性系视线。
3. 6D LOS 卡尔曼滤波估计 `lambda` 和 `lambda_dot`。
4. 检测框面积通道估计尺度膨胀和 TTC。
5. 输出 `g_eval = K(TTC) * lambda_dot_I` 作为导引评估量。

注意：`g_eval` 不是飞控控制指令，不会通过 MAVLink 发给真实飞控。

## 5. AirSim Blocks 场景要求

运行 `examples/run_airsim_blocks.py` 前，需要满足：

- Blocks 已经启动。
- AirSim RPC 能正常连接。
- 场景里存在两个 multirotor vehicle：
  - `Interceptor`
  - `Intruder`
- 入侵无人机的 mesh 名称能被 `Intruder*` 匹配。

如果你的 settings 或场景里车辆名字不同，需要通过命令行参数指定。

项目已提供一份 Blocks 配置示例：

```text

```

这份配置按官方 multi-vehicle 文档在 `Vehicles` 下显式定义 `Interceptor` 和 `Intruder`。AirSim 多机 API 调用必须传入对应的 `vehicle_name`，脚本当前也按这个规则分别控制两架机。

当前默认不是通过 settings 把飞机直接初始化到 50m，而是按正常位置生成，脚本启动后先执行预备爬升，再进入拦截循环。AirSim 使用 NED 坐标系，向上为负 `Z`，所以 50m 高度对应 `Z=-50`。

- settings 初始 `Interceptor`：`X=0, Y=0, Z=-2`
- settings 初始 `Intruder`：`X=35, Y=-20, Z=-2`
- 预备阶段：两架机先 `takeoffAsync()` 离地，再 `moveToZAsync(-50)` 爬升到 50m
- 爬升完成后：两架机 `hoverAsync()` 稳定，然后才开始视觉检测和拦截循环

相关参数：

- `--climb-to-altitude`：默认开启预备爬升
- `--no-climb-to-altitude`：关闭预备爬升，直接进入拦截循环
- `--intercept-altitude-m 50`：拦截开始高度，默认 50m
- `--climb-speed 5`：爬升速度，默认 5m/s
- `--climb-timeout-s 60`：爬升超时，默认 60s

可以用脚本安装到 AirSim 默认 settings 位置。安装时会自动备份原文件：

```bash

./install_airsim_blocks_settings.sh
```

安装完成后必须重启 Blocks，因为 AirSim 只在启动时读取 `~/Documents/AirSim/settings.json`。

也可以手动复制：

```bash

cp config/airsim_blocks_settings.json ~/Documents/AirSim/settings.json
```

检查当前 Blocks 里有哪些 vehicle：

```bash

```

正常启动 Blocks 后应看到：

```text
AirSim vehicles: 'Interceptor', 'Intruder'
```

如果只看到 `SimpleFlight` 或没有 `Intruder`，说明 Blocks 没有加载项目提供的 multi-vehicle settings，需要确认启动命令里带有 `-settings=/home/linux/Documents/PNG/config/airsim_blocks_settings.json`。

当前示例只使用 AirSim 内置检测返回的 `box2D`：

- 不使用 `simGetObjectPose("Intruder")`

## 8. AirSim 云台相机纯视觉 PNG 验证

当前新增一个独立验证脚本：

```bash
python3 examples/run_airsim_gimbal_vision_png.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

这个脚本用于验证“框架云台相机 + AirSim detect + 6D LOS + TTC”的纯视觉比例导引思路。它和最终交付程序不同，也和真值 PNG 基线不同：

- 导引输入只使用 AirSim detect 返回的 `box2D` 检测框。
- 不使用入侵无人机真实位置作为导引输入。
- 不使用入侵无人机真实速度作为导引输入。
- 两机爬升到指定高度后，默认允许读取一次入侵无人机真实位置，用于把云台预对准到目标；后续导引循环不再使用入侵机真值。
- 入侵无人机真实位置默认持续进入离线日志、命中判定和轨迹绘图，只作为算法评价依据；如需关闭可加 `--no-diagnostic-truth`。
- 默认不弹出 OpenCV 小窗口，以减少 `simGetImage()` 和本机显示开销，提高 AirSim 控制循环刷新率；需要人工观察检测框时加 `--show-window`。

### 8.1 云台相机视觉链路

每个周期执行：

1. 设置当前云台 yaw/pitch 到拦截机相机。
2. 调用 AirSim detect 获取入侵无人机 bbox。
3. 用 bbox center 误差更新云台，使目标保持在画面中心。
4. 用 bbox center、当前云台姿态和拦截机姿态计算惯性系 LOS。
5. 6D LOS 滤波输出 `lambda_I`、`lambda_dot_I`、`omega_los`。
6. bbox area 进入 TTC 尺度膨胀通道。
7. LOS 和 TTC 门控都通过时，输出：

```text
g_eval = K(TTC) * omega_los
```

8. 仿真中再把视觉导引评估量映射为 AirSim 速度命令。拦截机命令显式使用 `ForwardOnly`，并将 `yaw_mode` 设置为当前速度指令的绝对航向角，使机体机头跟随导引方向，而不是默认保持 `yaw_rate=0`。

启动时的预对准只发生一次：

```text
camera_offset_body = [camera_x, camera_y, camera_z]
p_camera = p_interceptor + R_IB * camera_offset_body
relative_body = R_IB^T * (p_intruder - p_camera)
gimbal_yaw = atan2(relative_body_y, relative_body_x)
gimbal_pitch = atan2(relative_body_z, horizontal_range)
```

这里的 `p_intruder` 只用于让目标初始进入画面中心，不进入后续 PNG 计算。当前默认相机外参为 `--camera-x 0 --camera-y 0 --camera-z -0.5`，AirSim 使用 NED 坐标，`camera_z=-0.5` 表示相机相对拦截无人机机体中心上移 0.5m，用于减少机架遮挡视角。

末端阶段如果出现 `bbox_clipped`、bbox 面积过大、云台接近限位，或连续检测后突然丢失，脚本会短时间进入 `blind_push`。当前不是锁死最后一帧，而是使用最近短窗口的三维 `v_cmd` 平均值，再叠加受限的 LOS 趋势偏置和 NED 向上的 `pitch_up` 速度偏置，避免末端视觉失效时立刻清零垂直通道。

云台版本还有单独的末端云台策略：`Tracking` 阶段正常用 bbox 中心误差更新云台；进入 `TerminalVisual` 后，云台更新增益按 `--terminal-gimbal-gain-scale` 降低，避免末端大框抖动把云台拉到限位；如果出现 `bbox_clipped` 或进入 `BlindPush`，默认冻结上一帧可靠云台 yaw/pitch，不再用裁切框继续追踪。这个逻辑只影响“下一帧相机看哪里”，LOS 去旋转仍使用当前帧曝光时的云台姿态。

当前又增加了终端像面 KF 外推层。它对 bbox center 转换得到的角误差 `theta_x/theta_y` 建立 4D 常速度卡尔曼滤波 `[theta_x, theta_y, theta_dot_x, theta_dot_y]`。当末端 bbox 被裁切或进入 `BlindPush` 时，如果 KF 仍有效，云台不会简单冻结，而是用 KF 预测出的角误差生成虚拟中心点继续短时扫动；超过 `--terminal-image-kf-max-predict-s` 后自动回退到 hold 逻辑。

可以用开关关闭该外推：

```bash
python3 examples/run_airsim_gimbal_vision_png.py --no-terminal-extrapolation
```

```bash

--terminal-soft-enter-area-ratio 0.05
--terminal-cutoff-area-ratio 0.60
--terminal-gimbal-limit-area-ratio 0.05
--terminal-cutoff-miss-count 3
--terminal-blind-duration-s 0.30
--terminal-command-average-window-s 0.10
--terminal-trend-bias-gain 0.10
--terminal-trend-bias-max-mps 1.5
```

### 8.2 输出文件

每次运行会在 `logs/` 下输出：

```text
gimbal_vision_png_*.csv
gimbal_vision_png_*_meta.json
gimbal_vision_png_*.png
```

CSV 重点字段：

- `pixel_error_x / pixel_error_y`：目标偏离画面中心的像素误差
- `gimbal_yaw_deg / gimbal_pitch_deg`：云台角度
- `los_quality / ttc_quality`
- `ttc / ttc_area / ttc_area_dot`
- `lambda_* / omega_* / g_eval_*`
- `v_cmd_*`
- `loop_dt / command_duration`：实际循环间隔和发送给 AirSim 的速度命令持续时间
- `guidance_mode`：`ttc_png` 表示 TTC-PNG 生效，`los_fallback` 表示 LOS 有效但 TTC 暂未通过时使用低增益 fallback，`invalid` 表示未输出导引修正
- `terminal_state / terminal_reason / terminal_area_ratio / terminal_miss_count`：末端状态机诊断
- `terminal_arm_source / terminal_cutoff_source`：终端武装和切断来源，例如 `valid_guidance`、`image_kf_soft`、`bbox_clipped`、`gimbal_limit`
- `terminal_profile / gimbal_update_enabled / gimbal_gain_scale / gimbal_hold_reason / gimbal_update_source`：云台末端适配器诊断，`gimbal_update_source=kf` 表示使用像面 KF 预测继续扫云台
- `image_kf_takeover_allowed / image_kf_takeover_reason`：像面 KF 是否接管以及接管门控原因
- `image_kf_valid / image_kf_mode / image_kf_theta_* / image_kf_theta_dot_* / image_kf_age_s / image_kf_quality / image_kf_reject_reason`：终端像面 KF 诊断
- `experiment_type / speed_ratio / intruder_speed / intruder_altitude_offset_m / speed_cap / fov_deg / camera_*`：实验配置快照，便于不同批次日志横向对比
- `blind_elapsed_s / blind_decay / blind_sample_count`：盲推持续时间、衰减和参与平均的样本数
- `v_cmd_base_* / v_cmd_trend_bias_* / v_cmd_pitch_up_bias_*`：盲推基准速度、LOS 趋势偏置和上仰偏置
- `range / horizontal_range / vertical_error`：仅在 `--diagnostic-truth` 打开时用于离线诊断
- `hit / collision_reason / *_collision_object`：成功只由 AirSim 报告 `Interceptor` 和 `Intruder` 发生 collision 决定，并且 `object_name` 必须按通配符匹配到另一架无人机

PNG 图包含：

- 两机 3D 轨迹

## 9. AirSim 捷联固定相机纯视觉 PNG 验证

当前新增一个独立捷联验证脚本：

```bash
python3 examples/run_airsim_strapdown_vision_png.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

这个脚本用于验证“固定前向相机 + AirSim detect + 6D LOS + TTC”的比例导引链路。它和云台脚本的关键差异是：

- 不动态调整相机 yaw/pitch。
- 相机固定在机体系前向，默认外参为 `--camera-x 0 --camera-y 0 --camera-z -0.5`。
- 两机爬升到起始高度后，默认允许读取一次入侵机真值，只用于把拦截机机头 yaw 对准目标，让目标一开始进入固定相机视场。
- 后续导引循环不使用入侵机真实位置或真实速度，只使用 detect 返回的 `box2D`。
- 用 bbox 横向误差生成有限幅的机体 yaw-rate 命令，使固定相机持续看向目标。
- 速度导引仍使用 3D LOS，保留高度方向 `v_cmd_z`。

捷联版本的末端逻辑和云台版本不同：它没有可冻结的云台，目标一旦接近视野边界，真正需要保持的是机体 yaw-rate 趋势。脚本会记录最近短窗口的 yaw-rate 指令；`TerminalVisual` 阶段按 `--terminal-yaw-rate-scale` 适当降幅，进入 `BlindPush` 后按 `--terminal-yaw-rate-decay-tau-s` 指数衰减保持短窗口平均值，避免检测突然丢失时 yaw-rate 立刻归零导致固定相机甩开目标。

如果终端像面 KF 有效，捷联版本会优先用 `theta_x` 和 `theta_dot_x` 合成 yaw-rate：`yaw_rate = yaw_error_gain * theta_x + theta_dot_x`，再按 `--max-yaw-rate-deg` 限幅。`theta_y` 暂时只写日志，不直接接 pitch 控制，避免和现有 3D `v_cmd_z` 产生双重垂直修正。

默认起始高度和云台脚本一致：

```text

Intruder:    80m altitude, AirSim NED Z=-80
```

```bash
--terminal-yaw-rate-decay-tau-s 0.18
--terminal-yaw-rate-scale 0.70

--terminal-image-kf-max-predict-s 0.35
```

CSV 中重点看：

- `pixel_error_x / pixel_error_y`：固定相机中目标偏心量。
- `yaw_rate_raw_deg_s`：bbox 横向误差直接生成的原始 yaw-rate。

## 10. AirSim 真值 PNG 基线验证

为了把“导引律/控制方式是否可行”和“纯视觉感知是否可靠”拆开验证，项目新增了一个真值比例导引基线脚本：

```bash
python3 examples/run_airsim_truth_png.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

这个脚本和纯视觉脚本不同：它会直接读取 AirSim 中两架无人机的真实位置和真实速度。

```text
simGetGroundTruthKinematics("Interceptor")
simGetGroundTruthKinematics("Intruder")
```

注意：AirSim 的 `simGetGroundTruthKinematics().position` 是相对每架车自身起点的局部坐标，不是统一世界坐标。脚本会读取 `config/airsim_blocks_settings.json` 中每架车的 `X/Y/Z` 起点偏移，并计算：

```text
world_position = vehicle_start_origin + local_ground_truth_position
```

这样两架无人机的位置才可以相减得到真实相对距离。如果使用了其他 settings 文件，需要通过 `--settings-path` 指向对应 JSON。

因此它只用于 AirSim 仿真上限验证，不是可部署的视觉感知方案，也不代表真实系统可以获得这些信息。

### 10.1 真值 PNG 的默认场景

默认参数：

- 拦截无人机：`Interceptor`
- 入侵无人机：`Intruder`
- 默认每次运行先 `reset()` AirSim，避免沿用上一轮失败后的末端位置；如需保留现场，可加 `--no-reset`
- 起始后先爬升到 `50 m` 高度，再进入拦截循环
- 爬升后先 hover 稳定，默认等待 `--settle-s 2`
- 入侵无人机速度：`5 m/s`
- 入侵无人机默认沿 AirSim NED 坐标 `+Y` 方向直线飞行
- 拦截无人机速度上限：`speed_ratio * intruder_speed`
- 默认 `speed_ratio=2`，即拦截无人机速度上限约 `10 m/s`
- 比例导引系数：`N=3.0`
- 最大加速度：`15 m/s^2`
- 成功判定：AirSim 报告 `Interceptor` 和 `Intruder` 两机发生 collision，且碰撞对象名必须匹配另一机通配符
- 控制频率：`20 Hz`
- 默认启用 AirSim 多旋翼垂直速度包装层：`--altitude-correction`

### 10.2 真值 PNG 的计算流程

每个控制周期执行：

```text
r = p_intruder - p_interceptor
v_rel = v_intruder - v_interceptor
lambda = r / |r|
Vc = -dot(v_rel, lambda)
omega_los = cross(r, v_rel) / |r|^2
a_cmd = N * Vc * cross(omega_los, lambda)
```

工程处理：

- `Vc <= 0` 时认为目标不再闭合，本周期不输出有效 PNG 加速度。
- `a_cmd` 会按 `--max-accel` 限幅。
- AirSim 多旋翼用速度接口控制，所以脚本把加速度做一步速度积分：

```text
v_cmd = v_current + a_cmd * dt
```

- `v_cmd` 再按 `speed_ratio * intruder_speed` 做速度上限约束。
- 为避免拦截机从悬停开始时没有前向速度，脚本默认保留一个速度保持项，近似模拟拦截器持续前飞。

### 10.3 真值 PNG 的输出

每次运行会在 `logs/` 下输出：

```text
truth_png_trajectory_*.csv
truth_png_trajectory_*_meta.json
truth_png_trajectory_*.png
```

CSV 记录：

- 两机真实位置和速度
- 相对距离 `range`
- 水平距离 `horizontal_range`
- 高度误差 `vertical_error`
- 闭合速度 `closing_speed`
- 视线方向 `los`
- 视线角速度 `omega_los`
- PNG 加速度 `a_cmd`
- AirSim 速度指令 `v_cmd`
- 是否发生双机碰撞 `hit`
- 碰撞来源 `collision_reason`

PNG 图片包含：

- 两机 3D 轨迹；如果本机 matplotlib 的 3D 投影不可用，会自动退回俯视 2D 轨迹
- 距离随时间变化曲线

脚本一旦检测到双机 collision，会立即停止循环，因此 CSV 和 PNG 都只保留碰撞前以及碰撞帧的数据，不再包含碰撞后的轨迹。默认只接受 `Intruder*` / `Interceptor*` 这类对象名匹配；如果 AirSim 返回的 mesh 名称不同，可以用 `--collision-intruder-pattern` 和 `--collision-interceptor-pattern` 指定多个通配符。

### 10.4 如何用真值 PNG 判断问题

推荐先运行真值 PNG：

```bash
python3 examples/run_airsim_truth_png.py --enable-motion --duration-s 60 --intruder-speed 5 --speed-ratio 2
```

如果真值 PNG 都无法拦截，优先检查：

- 日志第一行的 `v_cmd_z` 是否异常大；如果接近 `-8 m/s`，说明爬升残余速度进入了拦截阶段，应确认脚本已经启用 hover settle
- 如果 `horizontal_range` 已经接近 0，但 `range` 仍有 1-3m，说明主要是高度误差；保持 `--altitude-correction` 开启，或适当提高 `--vertical-kp` 和 `--vertical-speed-limit`
- 初始位置是否导致几何关系不适合当前速度比
- `--speed-ratio` 是否太低
- `--max-accel` 是否太低
- `--navigation-constant` 是否需要从 `3.0` 调到 `4.0` 或 `5.0`
- AirSim 速度控制是否跟得上
- 两机是否确实爬升到同一高度

如果真值 PNG 可以拦截，而纯视觉版本不能拦截，再重点检查：

- AirSim detect 是否稳定给出检测框
- 检测框中心是否跳动
- 6D LOS 滤波是否过度滞后
- TTC 面积膨胀通道是否失效
- 纯视觉脚本的横向速度限幅是否过小

```text

Waiting for TCP connection on port 4560
```

这不是 Vulkan 问题，而是 PX4 SITL 的 TCP 端口 `4560` 被占用。常见原因是已经启动过一个 Blocks/AirSim 实例，或者上一次 Blocks 崩溃后 PX4 与旧连接没有完全清理。

```bash
pgrep -af 'Blocks|px4|px4-simulator'
```

清理后重启：

```bash
pkill -f 'Blocks/Binaries/Linux/Blocks' || true
pkill -f 'px4_sitl_default|PX4-Autopilot.*px4|px4-simulator' || true

cd /home/linux/Documents/PNG
./run_px4_sitl.sh
./run_blocks_px4_sitl_single.sh
```

注意：PX4 终端要保持打开。不要同时启动两个使用 `TcpPort=4560` 的 Blocks 实例。

### 12.1 替换目标检测

### 12.2 调整 6D LOS 滤波

相关文件：

```text
vision_guidance/los_filter.py
```

重点参数：

- 过程噪声：影响 `lambda_dot` 的响应速度。
- 观测噪声：影响检测框中心抖动的抑制强度。
- 单位向量约束：保证 `lambda` 始终接近单位视线向量。

调参原则：

- 检测框中心抖动大时，提高观测噪声。
- 目标机动强时，提高过程噪声。
- 输出滞后明显时，适度降低平滑强度。

### 12.3 调整 TTC 和尺度膨胀

相关文件：

```text
vision_guidance/ttc.py
```

重点关注：

- 面积低通滤波强度。
- 面积变化率阈值。
- TTC 合理范围门限。

调参原则：

- 面积抖动大时，加大低通滤波。
- 末端响应太慢时，降低滤波滞后。
- 目标姿态变化导致面积突变时，需要更严格的质量门控。

### 12.4 调整 PNG 评估量

相关文件：

```text
vision_guidance/png_eval.py
```

当前输出是评估量：

```text
g_eval = K(TTC) * lambda_dot_I
```

它用于日志、回放和仿真监督，不直接等同于真实飞机的加速度设定点。

## 14. 安全边界

当前项目仅用于仿真、日志评估、算法回放和监督实验：

- 不发送 MAVLink。
- 不控制真实飞控。
- 纯视觉链路不读取入侵无人机真实位置作为算法输入。
- 纯视觉链路不读取入侵无人机真实速度作为算法输入。
- 真值 PNG 链路会读取 AirSim ground truth，但仅用于仿真基线验证。
- 不把 `g_eval` 当成真实控制指令。

如果后续要接入真实系统，需要单独增加硬件安全约束、地面急停、仿真到半实物验证流程、飞控模式限制和完整的飞行安全审查。

混合配置采用双机：

- `Interceptor`：`PX4Multirotor`，连接 PX4 SITL，控制闭环经过 PX4。
- `Intruder`：`SimpleFlight`，继续由 AirSim API 直接控制，用作可控目标机。

这样做的目的是验证拦截机的 PX4/EKF/IMU 控制链路，同时保留入侵机的可控运动。但 AirSim 源码中对 PX4 + SimpleFlight 混合多机存在兼容性提示，实际调试应先跑通单 PX4，再扩展混合或双 PX4 多机。完整双 PX4 多机需要两个 PX4 SITL 实例和不同端口，当前配置暂不默认启用。

### 15.3.1 方案 A：PX4 拦截机 + SimpleFlight 入侵机真值 PNG

方案 A 用于先验证 PX4 SITL 闭环下的拦截效果：

- PNG 算法内部暂时读取真值相对位置和速度，只用于验证 PX4 速度跟踪、过载响应、碰撞判定和日志链路。

终端 2：启动混合双机 Blocks：

该脚本会默认给 `examples/run_airsim_truth_png.py` 传入 `--px4-interceptor`，使用 PX4 友好的预备流程：SimpleFlight 入侵机按原逻辑爬升，PX4 拦截机通过连续 offboard 高度 setpoint 爬升并轮询高度状态。这样避免 PX4 在 `takeoffAsync().join()` 或 `moveToZAsync().join()` 上长时间卡住。

### 15.4 端口和关键配置

`config/airsim_blocks_px4_sitl_settings.json` 中拦截机关键配置：

```json
"VehicleType": "PX4Multirotor",
"UseTcp": true,
"LockStep": true,
"LocalHostIp": "127.0.0.1",
"ControlIp": "127.0.0.1",
"TcpPort": 4560,
"ControlPortLocal": 14540,
"ControlPortRemote": 14580
```

含义：

- `TcpPort=4560`：AirSim 与 PX4 SITL 仿真后端连接。
- `ControlPortLocal=14540`：AirSim 本地 MAVLink 控制端口。
- `ControlPortRemote=14580`：PX4 MAVLink 远端端口。
- `LockStep=true`：AirSim 和 PX4 按锁步时钟同步，减少仿真时间漂移。
- `LocalHostIp/ControlIp/UdpIp=127.0.0.1`：PX4 和 Blocks 都运行在本机时显式绑定 localhost，避免多网卡机器选错地址。

配置里还降低了气压计噪声：

```json
"Barometer": {
  "SensorType": 1,
  "Enabled": true,
  "PressureFactorSigma": 0.0001825
}
```

这是 AirSim 官方 PX4 SITL 文档推荐做法之一，有助于 EKF 高度估计稳定。

### 15.5 连接检查

Blocks 和 PX4 都启动后，检查 AirSim 车辆：

```bash

PY
```

正常应包含：

```text

```

再检查 PX4 拦截机是否能响应 API：

```bash
python3 - <<'PY'

c.confirmConnection()
c.enableApiControl(True, vehicle_name="Interceptor")
c.armDisarm(True, vehicle_name="Interceptor")
print(c.getMultirotorState(vehicle_name="Interceptor").landed_state)
PY
```

如果 `enableApiControl` 或 `armDisarm` 报错，先看 PX4 终端是否已经完成 EKF 初始化。PX4 控制台中通常需要看到 GPS/EKF 通过检查后，才适合起飞和切入 offboard 控制。

### 15.6 运行算法时的注意事项

PX4 SITL 下，拦截机不再是 SimpleFlight 理想控制器。实际表现会变化：

- 起飞、爬升、速度跟踪会有 PX4 控制器动态延迟。
- `kinematics_estimated` 更接近飞控估计链路，不再等价于 SimpleFlight 的理想真值反馈。
- 需要等待 EKF healthy、GPS fix 和 commander ready 后再进入拦截。
- 原来 SimpleFlight 可用的高加速度速度命令，在 PX4 下可能被姿态、速度、加速度、failsafe 或 offboard 频率限制。

当前视觉 PNG 程序主要是围绕 SimpleFlight 批量测试调过的；切到 PX4 后建议先做小速度、小距离、短时间验证，不要直接复用最高速工况。

### 15.7 常见问题

PX4 卡在等待 simulator：

- 确认启动 Blocks 使用的是 `./run_blocks_px4_sitl.sh`。
- 确认没有另一个进程占用 TCP 4560。
- 确认 settings 中 `Interceptor` 是 `PX4Multirotor`。

AirSim 能连接但 PX4 不起飞：

- 等待 PX4 输出 `Ready for takeoff`。
- 检查 PX4 是否有 preflight check fail。
- 检查 AirSim origin geopoint、GPS、气压计是否正常。

控制指令不稳定：

- 降低拦截速度和 `speed-ratio`。
- 提高控制循环频率，确保 offboard 指令持续发送。
- 先用真值 PNG 基线验证 PX4 能否跟踪速度，再切换视觉 PNG。

需要双 PX4 多机：

- 为 `Intruder` 再启动一个 PX4 SITL 实例。
- 给第二架机配置不同的 `TcpPort`、`ControlPortLocal`、`ControlPortRemote`。
- 当前项目先不默认启用该模式，以避免端口和多实例初始化复杂度影响单机 PX4 验证。
