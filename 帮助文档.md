# PNG 项目后续运行帮助文档

本文档说明当前项目在本机继续运行、验证和调试的流程。当前代码定位是纯视觉比例导引算法的仿真评估框架：目标检测先由 AirSim 内置 `detect` 接口提供，算法只使用检测框，不读取入侵无人机真实位置、真实速度或相对真值。

## 1. 当前项目内容

项目根目录：`/home/linux/Documents/PNG`

主要文件：

- `vision_guidance/`：纯视觉导引算法模块。
- `examples/run_synthetic.py`：不依赖 AirSim 的合成检测框示例。
- `examples/run_airsim_blocks.py`：AirSim Blocks 示例，使用 AirSim 内置检测框。
- `tests/`：单元测试。
- `run_blocks_nvidia.sh`：强制使用 NVIDIA 显卡启动 Blocks 的脚本。
- `拦截方案.md`：理论方案文档。
- `README.md`：英文简要说明。

当前实现的核心链路：

1. 检测框中心点 `(u, v)` 转换为相机视线向量。
2. 结合图像曝光时间对应的姿态矩阵，转换到惯性系视线。
3. 6D LOS 卡尔曼滤波估计 `lambda` 和 `lambda_dot`。
4. 检测框面积通道估计尺度膨胀和 TTC。
5. 输出 `g_eval = K(TTC) * lambda_dot_I` 作为导引评估量。

注意：`g_eval` 不是飞控控制指令，不会通过 MAVLink 发给真实飞控。

```text

```

当前默认不是通过 settings 把飞机直接初始化到 50m，而是按正常位置生成，脚本启动后先执行预备爬升，再进入拦截循环。AirSim 使用 NED 坐标系，向上为负 `Z`，所以 50m 高度对应 `Z=-50`。

相关参数：

- `--climb-to-altitude`：默认开启预备爬升
- `--no-climb-to-altitude`：关闭预备爬升，直接进入拦截循环
- `--intercept-altitude-m 50`：拦截开始高度，默认 50m
- `--climb-speed 5`：爬升速度，默认 5m/s
- `--climb-timeout-s 60`：爬升超时，默认 60s

## 8. AirSim 云台相机纯视觉 PNG 验证

当前新增一个独立验证脚本：

```bash
python3 examples/run_airsim_gimbal_vision_png.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

这个脚本用于验证“框架云台相机 + AirSim detect + 6D LOS + TTC”的纯视觉比例导引思路。它和最终交付程序不同，也和真值 PNG 基线不同：

- 导引输入只使用 AirSim detect 返回的 `box2D` 检测框。
- 不使用入侵无人机真实位置作为导引输入。
- 不使用入侵无人机真实速度作为导引输入。
- 两机爬升到指定高度后，默认允许读取一次入侵无人机真实位置，用于把云台预对准到目标；后续导引循环不再使用入侵机真值。
- 入侵无人机真实位置默认持续进入离线日志、命中判定和轨迹绘图，只作为算法评价依据；如需关闭可加 `--no-diagnostic-truth`。
- 默认不弹出 OpenCV 小窗口，以减少 `simGetImage()` 和本机显示开销，提高 AirSim 控制循环刷新率；需要人工观察检测框时加 `--show-window`。

### 8.1 云台相机视觉链路

每个周期执行：

1. 设置当前云台 yaw/pitch 到拦截机相机。
2. 调用 AirSim detect 获取入侵无人机 bbox。
3. 用 bbox center 误差更新云台，使目标保持在画面中心。
4. 用 bbox center、当前云台姿态和拦截机姿态计算惯性系 LOS。
5. 6D LOS 滤波输出 `lambda_I`、`lambda_dot_I`、`omega_los`。
6. bbox area 进入 TTC 尺度膨胀通道。
7. LOS 和 TTC 门控都通过时，输出：

```text
g_eval = K(TTC) * omega_los
```

8. 仿真中再把视觉导引评估量映射为 AirSim 速度命令。拦截机命令显式使用 `ForwardOnly`，并将 `yaw_mode` 设置为当前速度指令的绝对航向角，使机体机头跟随导引方向，而不是默认保持 `yaw_rate=0`。

启动时的预对准只发生一次：

```text
camera_offset_body = [camera_x, camera_y, camera_z]
p_camera = p_interceptor + R_IB * camera_offset_body
relative_body = R_IB^T * (p_intruder - p_camera)
gimbal_yaw = atan2(relative_body_y, relative_body_x)
gimbal_pitch = atan2(relative_body_z, horizontal_range)
```

这里的 `p_intruder` 只用于让目标初始进入画面中心，不进入后续 PNG 计算。当前默认相机外参为 `--camera-x 0 --camera-y 0 --camera-z -0.5`，AirSim 使用 NED 坐标，`camera_z=-0.5` 表示相机相对拦截无人机机体中心上移 0.5m，用于减少机架遮挡视角。

末端阶段如果出现 `bbox_clipped`、bbox 面积过大、云台接近限位，或连续检测后突然丢失，脚本会短时间进入 `blind_push`，保留上一帧完整三维 `v_cmd`，包括 `v_cmd_z`，避免末端视觉失效时立刻清零垂直通道。

### 8.2 输出文件

每次运行会在 `logs/` 下输出：

```text
gimbal_vision_png_*.csv
gimbal_vision_png_*.png
```

CSV 重点字段：

- `pixel_error_x / pixel_error_y`：目标偏离画面中心的像素误差
- `gimbal_yaw_deg / gimbal_pitch_deg`：云台角度
- `los_quality / ttc_quality`
- `ttc / ttc_area / ttc_area_dot`
- `lambda_* / omega_* / g_eval_*`
- `v_cmd_*`
- `loop_dt / command_duration`：实际循环间隔和发送给 AirSim 的速度命令持续时间
- `guidance_mode`：`ttc_png` 表示 TTC-PNG 生效，`los_fallback` 表示 LOS 有效但 TTC 暂未通过时使用低增益 fallback，`invalid` 表示未输出导引修正
- `range / horizontal_range / vertical_error / hit`：仅在 `--diagnostic-truth` 打开时用于离线诊断

PNG 图包含：

- 两机 3D 轨迹
- range 曲线
- 像素误差曲线
- 云台 yaw/pitch 曲线

## 9. AirSim 真值 PNG 基线验证

为了把“导引律/控制方式是否可行”和“纯视觉感知是否可靠”拆开验证，项目新增了一个真值比例导引基线脚本：

```bash
python3 examples/run_airsim_truth_png.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

这个脚本和纯视觉脚本不同：它会直接读取 AirSim 中两架无人机的真实位置和真实速度。

```text
simGetGroundTruthKinematics("Interceptor")
simGetGroundTruthKinematics("Intruder")
```

注意：AirSim 的 `simGetGroundTruthKinematics().position` 是相对每架车自身起点的局部坐标，不是统一世界坐标。脚本会读取 `config/airsim_blocks_settings.json` 中每架车的 `X/Y/Z` 起点偏移，并计算：

```text
world_position = vehicle_start_origin + local_ground_truth_position
```

这样两架无人机的位置才可以相减得到真实相对距离。如果使用了其他 settings 文件，需要通过 `--settings-path` 指向对应 JSON。

因此它只用于 AirSim 仿真上限验证，不是可部署的视觉感知方案，也不代表真实系统可以获得这些信息。

### 9.1 真值 PNG 的默认场景

默认参数：

- 拦截无人机：`Interceptor`
- 入侵无人机：`Intruder`
- 默认每次运行先 `reset()` AirSim，避免沿用上一轮失败后的末端位置；如需保留现场，可加 `--no-reset`
- 起始后先爬升到 `50 m` 高度，再进入拦截循环
- 爬升后先 hover 稳定，默认等待 `--settle-s 2`
- 入侵无人机速度：`5 m/s`
- 入侵无人机默认沿 AirSim NED 坐标 `+Y` 方向直线飞行
- 拦截无人机速度上限：`speed_ratio * intruder_speed`
- 默认 `speed_ratio=2`，即拦截无人机速度上限约 `10 m/s`
- 比例导引系数：`N=3.0`
- 最大加速度：`15 m/s^2`
- 命中半径：`1.0 m`
- 控制频率：`20 Hz`
- 默认启用 AirSim 多旋翼垂直速度包装层：`--altitude-correction`

### 9.2 真值 PNG 的计算流程

每个控制周期执行：

```text
r = p_intruder - p_interceptor
v_rel = v_intruder - v_interceptor
lambda = r / |r|
Vc = -dot(v_rel, lambda)
omega_los = cross(r, v_rel) / |r|^2
a_cmd = N * Vc * cross(omega_los, lambda)
```

工程处理：

- `Vc <= 0` 时认为目标不再闭合，本周期不输出有效 PNG 加速度。
- `a_cmd` 会按 `--max-accel` 限幅。
- AirSim 多旋翼用速度接口控制，所以脚本把加速度做一步速度积分：

```text
v_cmd = v_current + a_cmd * dt
```

- `v_cmd` 再按 `speed_ratio * intruder_speed` 做速度上限约束。
- 为避免拦截机从悬停开始时没有前向速度，脚本默认保留一个速度保持项，近似模拟拦截器持续前飞。

### 9.3 真值 PNG 的输出

每次运行会在 `logs/` 下输出：

```text
truth_png_trajectory_*.csv
truth_png_trajectory_*.png
```

CSV 记录：

- 两机真实位置和速度
- 相对距离 `range`
- 水平距离 `horizontal_range`
- 高度误差 `vertical_error`
- 闭合速度 `closing_speed`
- 视线方向 `los`
- 视线角速度 `omega_los`
- PNG 加速度 `a_cmd`
- AirSim 速度指令 `v_cmd`
- 是否命中 `hit`

PNG 图片包含：

- 两机 3D 轨迹；如果本机 matplotlib 的 3D 投影不可用，会自动退回俯视 2D 轨迹
- 距离随时间变化曲线

### 9.4 如何用真值 PNG 判断问题

推荐先运行真值 PNG：

```bash
python3 examples/run_airsim_truth_png.py --enable-motion --duration-s 60 --intruder-speed 5 --speed-ratio 2
```

如果真值 PNG 都无法拦截，优先检查：

- 日志第一行的 `v_cmd_z` 是否异常大；如果接近 `-8 m/s`，说明爬升残余速度进入了拦截阶段，应确认脚本已经启用 hover settle
- 如果 `horizontal_range` 已经接近 0，但 `range` 仍有 1-3m，说明主要是高度误差；保持 `--altitude-correction` 开启，或适当提高 `--vertical-kp` 和 `--vertical-speed-limit`
- 初始位置是否导致几何关系不适合当前速度比
- `--speed-ratio` 是否太低
- `--max-accel` 是否太低
- `--navigation-constant` 是否需要从 `3.0` 调到 `4.0` 或 `5.0`
- AirSim 速度控制是否跟得上
- 两机是否确实爬升到同一高度

如果真值 PNG 可以拦截，而纯视觉版本不能拦截，再重点检查：

- AirSim detect 是否稳定给出检测框
- 检测框中心是否跳动
- 6D LOS 滤波是否过度滞后
- TTC 面积膨胀通道是否失效
- 纯视觉脚本的横向速度限幅是否过小

## 10. 常见问题

### 10.1 Blocks.sh not found

报错类似：

```text
Blocks.sh not found: ...
Set BLOCKS_DIR to the directory that contains Blocks.sh.
```

处理：

1. 确认 Blocks 已放在项目目录：

```bash
ls /home/linux/Documents/PNG/Blocks
```

2. 查找 `Blocks.sh`：

```bash
find /home/linux/Documents/PNG/Blocks -name Blocks.sh
```

3. 如果路径不是默认路径，用 `BLOCKS_DIR` 指定：

```bash

```

### 10.2 Vulkan VK_ERROR_DEVICE_LOST

这通常和混合显卡、Vulkan ICD 选择或驱动有关。

优先使用项目脚本启动：

```bash
./run_blocks_nvidia.sh
```

脚本会设置：

```bash
__NV_PRIME_RENDER_OFFLOAD=1

__VK_LAYER_NV_optimus=NVIDIA_only
VK_ICD_FILENAMES=/usr/share/vulkan/icd.d/nvidia_icd.json
VK_DRIVER_FILES=/usr/share/vulkan/icd.d/nvidia_icd.json
```

如果仍然崩溃，先确认 NVIDIA 驱动可用：

```bash
nvidia-smi
```

### 10.3 OpenXR 或 HMD warning

可能看到类似：

```text
Failed to find default runtime
Failed to enumerate extensions
```

当前脚本默认加了 `-nohmd`。如果 Blocks 正常打开，这些 warning 可以忽略。

### 10.4 AirSim Python 包缺失

报错类似：

```text
Install the AirSim Python package before running this example.
```

处理：

```bash
python3 -m pip install airsim==1.8.1
```

### 10.5 AirSim 连接失败

先确认 Blocks 已完全启动，再运行：

```bash
python3 examples/run_airsim_blocks.py --duration-s 30
```

如果仍无法连接，检查 AirSim settings、RPC 端口和 Blocks 是否卡在启动界面。

### 10.5.1 enableApiControl 抛 rpclib exception

报错类似：

```text
rpclib: function 'enableApiControl' ... threw an exception
```

最常见原因是当前 AirSim 场景里没有加载项目的多机 settings。当前脚本默认要求：

```text
Interceptor
Intruder
```

推荐使用项目提供的 `config/airsim_blocks_settings.json` 启动 Blocks，它会在 `Vehicles` 下显式创建 `Interceptor` 和 `Intruder`。

处理：

```bash
cd /home/linux/Documents/PNG
./install_airsim_blocks_settings.sh
```

然后关闭并重新启动 Blocks：

```bash
./run_blocks_nvidia.sh
```

另开终端确认车辆名：

```bash
python3 examples/run_airsim_blocks.py --list-vehicles
```

正常应看到：

```text
AirSim vehicles: 'Interceptor', 'Intruder'
```

随后直接运行正式示例：

```bash
python3 examples/run_airsim_blocks.py --enable-motion --intruder-speed 5 --speed-ratio 2
```

### 10.6 一直没有检测框

现象：

```text
detected=0
```

检查项：

- `--mesh` 是否匹配入侵无人机 mesh 名称。
- `--camera` 是否是拦截机当前使用的相机。
- 入侵无人机是否在相机视场内。
- 检测半径是否足够，默认 `20000 cm`。

可以尝试：

```bash
python3 examples/run_airsim_blocks.py --mesh "*Intruder*" --duration-s 30
```

## 11. 后续开发入口

### 11.1 替换目标检测

真实检测器或 YOLO/ByteTrack 后续只需要输出：

```text
FrameDetection:
  frame_id
  exposure_ts
  bbox_xyxy
  track_id
  score
```

对应定义在：

```text
vision_guidance/types.py
```

只要能稳定提供检测框中心、面积、时间戳和 track id，就可以接入当前纯视觉导引链路。

### 11.2 调整 6D LOS 滤波

相关文件：

```text
vision_guidance/los_filter.py
```

重点参数：

- 过程噪声：影响 `lambda_dot` 的响应速度。
- 观测噪声：影响检测框中心抖动的抑制强度。
- 单位向量约束：保证 `lambda` 始终接近单位视线向量。

调参原则：

- 检测框中心抖动大时，提高观测噪声。
- 目标机动强时，提高过程噪声。
- 输出滞后明显时，适度降低平滑强度。

### 11.3 调整 TTC 和尺度膨胀

相关文件：

```text
vision_guidance/ttc.py
```

重点关注：

- 面积低通滤波强度。
- 面积变化率阈值。
- TTC 合理范围门限。

调参原则：

- 面积抖动大时，加大低通滤波。
- 末端响应太慢时，降低滤波滞后。
- 目标姿态变化导致面积突变时，需要更严格的质量门控。

### 11.4 调整 PNG 评估量

相关文件：

```text
vision_guidance/png_eval.py
```

当前输出是评估量：

```text
g_eval = K(TTC) * lambda_dot_I
```

它用于日志、回放和仿真监督，不直接等同于真实飞机的加速度设定点。

## 13. 安全边界

当前项目仅用于仿真、日志评估、算法回放和监督实验：

- 不发送 MAVLink。
- 不控制真实飞控。
- 纯视觉链路不读取入侵无人机真实位置作为算法输入。
- 纯视觉链路不读取入侵无人机真实速度作为算法输入。
- 真值 PNG 链路会读取 AirSim ground truth，但仅用于仿真基线验证。
- 不把 `g_eval` 当成真实控制指令。

如果后续要接入真实系统，需要单独增加硬件安全约束、地面急停、仿真到半实物验证流程、飞控模式限制和完整的飞行安全审查。
